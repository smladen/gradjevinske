# CEOP construction-permit scraper

Scrapes construction permits from Serbia's public **Централна евиденција обједињених
процедура** (CEOP) registry at <https://ceop.apr.gov.rs> into a local SQLite
database. Built to run repeatedly — results are de-duplicated, so re-runs only add
new cases and refresh statuses of existing ones.

Currently configured for: **Novi Sad** (placeId `802824`), **all investors**,
cases created **on/after 2015-01-01**, stopping after **10,000 permits downloaded** per run.

## Install

```bash
pip install -r requirements.txt
playwright install chromium
```

## Run

```bash
python ceop_scraper.py
```

Output goes to `ceop_permits.db` (table `permits`). Inspect it with any SQLite
tool, e.g.:

```bash
sqlite3 ceop_permits.db "SELECT case_number, status, submission_type, created_date FROM permits ORDER BY created_date DESC;"
```

## How it works

The CEOP site is a single-page app whose search API
(`GET /ceopapi/api/search/search`) requires a Google **reCAPTCHA v3** token in a
`Recaptcha` request header on every call. Those tokens can only be generated by
Google's script running on the real page, so the scraper drives a headless
Chromium (via Playwright): it loads the CEOP page, mints a fresh token for each
request, and calls the API from inside the page context. Results are paginated
10 per page (the API's hard cap) and written to SQLite.

## Configuration

Edit the `CONFIG` dict at the top of `ceop_scraper.py`:

| Key | Meaning |
| --- | --- |
| `place_ids` | List of place IDs (the construction site location). `802824` = Novi Sad. |
| `investor` | Investor name, plain Latin text. Auto-transliterated to Cyrillic and sent in both scripts. Empty `""` = all investors. |
| `created_date_from` | Keep only cases with `CreatedDate` on/after this date (`YYYY-MM-DD`). |
| `search_string` | Optional partial case number, e.g. `ROP-NSD`. |
| `street` | Optional street name of the construction site. |
| `submission_number` | Optional exact submission number. |
| `db_path` | SQLite file path. |
| `headful` | `True` shows the browser window. Try this if results come back empty (reCAPTCHA score). |
| `request_delay_sec` | Delay between page requests (politeness). |
| `max_records` | Stop after this many permits are downloaded in a run; `0` = unlimited. |
| `geocode` | `True` = after scraping, geocode new addresses into `geocode_cache` (uses `geocode_permits.py`). `False` = skip. |
| `geocode_city` | City appended to each address when geocoding (default `Novi Sad, Serbia`). |

### Finding other place IDs

To scrape a different city, open the site, click **Напредна претрага** (Advanced
search), type a place into the **Место** field and select it, add another filter
(e.g. an investor) so the **Претражи** button enables, run the search, then read
the `placeIds` value from the request URL in your browser's DevTools → Network tab.

## Notes & caveats

- **Date meaning.** The stored `created_date` is when the case was created in the
  registry. CEOP only launched in 2016, so "after 2015" effectively includes the
  whole registry. There is no separate reliable "permit issued/decision date" in
  the list API; the per-case decision date lives in each case's detail page.
- **Permit type.** The registry mixes submission types (location conditions,
  building permits, work notifications, use permits, …). The `submission_type`
  column lets you filter to building permits (`Грађевинска дозвола`) in SQL if
  you only want those.
- **Broad location-only searches are blocked.** The registry disables search when
  only a place is given ("fill in more filters"), so a non-empty investor / street /
  case-number filter is required alongside the location. The current config relies
  on the `Graviton` investor filter.
- **reCAPTCHA.** If the site tightens bot protection, headless requests may start
  returning empty; set `headful = True` to run with a visible browser.
- This scrapes only public registry data and makes ordinary paginated requests
  with a polite delay.

## Database schema

```
permits(
  submission_id   TEXT PRIMARY KEY,   -- registry GUID, used for de-duplication
  case_number     TEXT,               -- e.g. ROP-NSD-43206-CPI-3/2025
  status          TEXT,
  submission_type TEXT,
  organization    TEXT,               -- competent authority
  address         TEXT,
  investors       TEXT,
  created_date    TEXT,
  pre_submit_date TEXT,
  place_ids       TEXT,
  first_seen_at   TEXT,               -- first run that saw this case
  last_seen_at    TEXT                -- most recent run that saw it
)
```

# Map & search app

An interactive Streamlit app that reads the scraped database, places permits on a
map, and lets you search by investor, location, date, status and type.

Submissions are **grouped by building**: the case-number prefix (first three
segments, e.g. `ROP-NSD-36653`) identifies one building, so the many requests an
investor files for the same building — `ROP-NSD-26138-CCF-6/2021`,
`ROP-NSD-26138-COFS-10/2022`, … — collapse into a single map marker and table row
(`ROP-NSD-26138`).

## Files

| File | Purpose |
| --- | --- |
| `permits_app.py` | The Streamlit app (map, filters, tables, downloads). |
| `permits_core.py` | Data logic: load, group, filter, aggregate, build the JSON. |
| `geocode_permits.py` | One-off geocoder: turns addresses into map coordinates. |
| `requirements-app.txt` | Dependencies for the app (separate from the scraper). |
| `dashboard_data.json` | Auto-written documented export of all calculated figures. |

## Setup & run

```bash
pip install -r requirements-app.txt

# 1) Add coordinates. The scraper already does this automatically after each run
#    (CONFIG["geocode"]); run this manually only to (re)geocode on demand.
python geocode_permits.py

# 2) Launch the app
streamlit run permits_app.py
```

The geocoder uses OpenStreetMap Nominatim at ~1 request/second and caches every
result in a `geocode_cache` table, so it only looks up each address once. The
scraper (`ceop_scraper.py`) calls this same step at the end of a run, so new
permits are geocoded as they are added; `geocode_permits.py` remains available to
run the step on its own. Permits without coordinates still appear in search and
tables — just not on the map.

## What the app shows

- **Map** — one clustered marker per building; click for the address, investor,
  submission count, date span, statuses and all case numbers in the group.
- **Filters** (sidebar) — investor, location/address, case number/group, status,
  submission type, and a created-date range.
- **Tables** — a building-groups summary and an expandable list of individual permits.
- **Downloads** — the documented `dashboard_data.json`, plus groups and filtered
  permits as CSV.

## dashboard_data.json

Per the analytics convention, all calculated figures are written to a separate
JSON file, and every number is explained in its `_documentation` section
(totals, counts by status / type / year, top investors, and per-building group
details with coordinates and date spans). The file reflects the filters active
when it was exported.
